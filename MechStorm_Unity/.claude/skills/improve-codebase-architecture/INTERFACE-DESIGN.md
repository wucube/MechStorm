# 接口设计

当用户想为选定的加深候选探索替代接口时，使用此并行子 agent 模式。基于"Design It Twice"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**（模块）、**interface**（接口）、**seam**（接缝）、**adapter**（适配器）、**leverage**（杠杆）。

## 流程

### 1. 框定问题空间

在生成子 agent 之前，为选定的候选写一段面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖的依赖，以及它们属于哪个类别（参见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的示意性代码草图来使约束具体化——不是提案，只是让约束变得具体的方式

将此展示给用户，然后立即进入步骤 2。用户在子 agent 并行工作时阅读和思考。

### 2. 生成子 agent

使用 Agent 工具并行生成 3+ 个子 agent。每个必须为加深模块产出一个**根本不同的**接口。

用独立的技术简报提示每个子 agent（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝后面是什么）。简报独立于步骤 1 中面向用户的问题空间说明。给每个 agent 不同的设计约束：

- Agent 1："最小化接口——目标是最多 1–3 个入口点。最大化每个入口点的杠杆。"
- Agent 2："最大化灵活性——支持多种用例和扩展。"
- Agent 3："为最常见调用者优化——让默认情况变得简单。"
- Agent 4（如适用）："围绕端口与适配器设计，用于跨接缝依赖。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，使每个子 agent 的命名与架构语言和项目领域语言保持一致。

每个子 agent 输出：

1. 接口（类型、方法、参数——以及不变量、顺序、错误模式）
2. 使用示例，展示调用者如何使用它
3. 实现在接缝后面隐藏了什么
4. 依赖策略和适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——杠杆高的地方，杠杆薄的地方

### 3. 呈现和比较

按顺序呈现设计，让用户可以消化每一个，然后用文字进行比较。按**深度**（接口处的杠杆）、**局部性**（变更集中的位置）和**接缝放置**进行对比。

比较后，给出你自己的建议：你认为哪个设计最强，为什么。如果不同设计的元素可以很好地组合，提出混合方案。要有主见——用户想要的是强有力的判断，不是菜单。
