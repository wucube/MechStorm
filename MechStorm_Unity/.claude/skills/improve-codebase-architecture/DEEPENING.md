# Deepening

如何在给定依赖的情况下安全地加深一组浅层模块。假设已了解 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**（模块）、**interface**（接口）、**seam**（接缝）、**adapter**（适配器）。

## 依赖分类

在评估加深候选时，对其依赖进行分类。类别决定了加深模块如何在其接缝处进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。始终可加深——合并模块并直接通过新接口测试。无需适配器。

### 2. 本地可替代（Local-substitutable）

拥有本地测试替身的依赖（PGLite 替代 Postgres、内存文件系统替身）。如果替身存在则可加深。加深模块在测试套件中使用替身运行。接缝是内部的；模块的外部接口处没有端口。

### 3. 远程但自有（Ports & Adapters）

跨网络边界的自有服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有逻辑；传输作为**适配器**注入。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

建议形式：*"在接缝处定义端口，为生产环境实现 HTTP 适配器，为测试实现内存适配器，这样逻辑位于一个深模块中，即使它跨网络部署。"*

### 4. 真正外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。加深模块将外部依赖作为注入端口接收；测试提供 mock 适配器。

## 接缝纪律

- **一个适配器意味着假想的接缝。两个适配器意味着真正的接缝。** 除非至少有两个适配器被证明合理（通常是生产 + 测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深模块可以有内部接缝（对其实现私有，由其自身测试使用）以及其接口处的外部接缝。不要仅仅因为测试使用内部接缝就通过接口暴露它们。

## 测试策略：替换，而非叠加

- 一旦加深模块接口处的测试存在，浅层模块上的旧单元测试就变成废品——删除它们。
- 在加深模块的接口处编写新测试。**接口就是测试面**。
- 测试通过接口断言可观察的结果，而非内部状态。
- 测试应在内部重构后存活——它们描述行为，而非实现。如果实现变更时测试也必须变更，那测试就越过了接口。
