# 工程协作与编码原则

## 1. 文档定位

本文定义 MechStorm 项目中通用的架构分析、编码、沟通和执行原则，适用于参与项目的开发者与 AI 编码助手。

本文只规定跨模块通用的工程协作方式，不重复具体框架 API、战斗架构和当前任务要求。发生冲突时按以下优先级执行：

1. 安全要求和用户当前明确指令。
2. `CLAUDE.md` 及项目技术规范。
3. `DEVELOPMENT_PROGRESS.md` 当前任务、验收标准和决策记录。
4. `BATTLE_ARCHITECTURE_ROADMAP.md` 等领域架构文档。
5. 本文的通用工程原则。

## 2. 核心结论

所有工作遵循以下总原则：

- 架构分析从问题本质和约束出发，不从现成模式或框架名称出发。
- 实现优先选择满足当前验收标准的最简单方案。
- 修改范围紧贴需求，不主动扩大任务边界。
- 输出先给结论，再给依据、方案、风险和验证结果。
- 多步任务以可验证目标驱动，持续执行到验收通过或明确受阻。

## 3. 架构与设计原则

### 3.1 第一性原理

分析架构、模块或代码组合时，先回答：

1. 要解决的真实问题是什么。
2. 当前必须满足哪些业务和技术约束。
3. 哪些状态、行为和依赖是不可省略的。
4. 哪些复杂度来自当前需求，哪些只是对未来的猜测。
5. 最小可验证方案是什么。

不得仅因为某个模式、框架或参考项目流行，就默认引入。外部方案只能作为证据和备选，最终设计必须映射到 MechStorm 的项目语言、当前阶段和验收标准。

### 3.2 DRY、KISS、SOLID 与 YAGNI

- **DRY**：消除已经出现且语义一致的重复，不为可能出现的重复提前抽象。
- **KISS**：优先使用直接、可读、可测试的实现，避免无必要的层级和间接调用。
- **SOLID**：按职责和变化原因拆分模块，依赖稳定契约，不机械追求接口数量。
- **YAGNI**：需求未要求、当前验收不依赖、近期没有明确使用方的能力不实现。

原则冲突时，优先保证正确性和可理解性，其次保证当前可维护性，最后才考虑未来扩展性。

### 3.3 规模约束

- 单个函数原则上不超过 50 行。超过时先判断是否混合了多个职责，再按语义拆分。
- 单个类或代码文件超过 500 行时，必须识别其职责、依赖和变化原因，评估是否需要拆分。
- 不能只为满足行数机械拆文件。拆分后的对象必须拥有清楚职责和稳定边界。
- 一次性流程不建立通用抽象；只有出现真实复用或独立变化原因时才提取。

## 4. 编码规范

### 4.1 遵循现有代码

- 编码前先读取相关目录、相邻实现、测试和项目规范。
- 复用项目已有库、模式和命名，不擅自引入新依赖。
- 与周边代码保持一致，除非现有写法违反项目硬约束或会直接导致缺陷。
- 使用有意义的类型、变量和函数名称，让命名承担主要说明职责。

### 4.2 注释与错误处理

- 只为复杂意图、关键约束和反直觉行为添加注释。
- 不添加重复代码字面含义的注释。
- 外部输入必须校验，不能假设其天然可信。
- 错误必须显式返回、抛出或记录，不允许静默吞掉。
- 密钥、密码、API Key 等敏感信息必须来自环境变量或安全配置，不得硬编码或写入日志。

### 4.3 精准修改

- 每一行差异都必须对应当前需求、必要修复或本次改动产生的清理。
- 不顺手格式化、重命名或重构无关代码。
- 发现无关死代码时只报告，不擅自删除。
- 本次改动产生的无用导入、变量、函数和临时文件必须清理。
- 未经明确要求，不删除、覆盖或移动用户未跟踪文件。

## 5. 编码前思考

开始实现前必须完成以下判断：

1. 需求是否清楚且可验证。
2. 是否存在多种会显著影响结果的合理理解。
3. 是否有更简单的现有实现或项目能力可以复用。
4. 方案是否引入新的技术债、依赖或维护成本。
5. 实现是否超出当前任务和阶段边界。

如果关键需求存在歧义，应给出互斥选项并请求用户选择，不得默默替用户做高影响决策。

如果用户方案会明显增加技术债、违反项目硬约束或存在更简单可靠的方案，应先说明结论、原因和替代方案。

## 6. 极简优先

实现前和交付前分别进行一次极简检查：

- 是否加入了需求未指定的扩展点、配置项或兼容层。
- 是否为尚未出现的场景增加了抽象。
- 是否可以删除一层包装而不损失语义。
- 是否可以复用已有对象而不是创建平行体系。
- 是否用 50 行能够清楚完成，却写成了 200 行。

如果答案表明实现绕得没有必要，应缩减到满足当前验收标准的最小方案。

## 7. 目标驱动执行

### 7.1 将指令转为验收目标

多步任务开始时，将需求转换为可验证目标：

```text
目标
-> 实现步骤
-> 每步验证方式
-> 最终验收条件
```

例如修复缺陷时，优先建立可复现测试，再修复实现并让测试通过；新增功能时，先确认行为边界和验收用例，再实现最小闭环。

### 7.2 持续执行

- 多步任务先建立计划，每一步都绑定验证点。
- 完成修改后运行项目规定的测试、类型检查、编译或静态检查。
- 验证失败时继续定位和修复，不以“代码已写完”作为完成标准。
- 只有全部验收通过，或存在必须由用户处理的明确阻塞时，才结束任务。
- 阻塞报告必须包含结论、原因、已完成内容和下一步解除方式。

## 8. 沟通规范

所有分析、方案、汇报、Issue 和 PR 描述遵循金字塔原理：

1. 先给结论或目的。
2. 再给背景和关键依据。
3. 然后说明方案或改动点。
4. 最后说明影响、风险和验证结果。

结构化内容应尽量按互斥且穷尽的方式分组，避免同一信息在多个章节重复。

用户提供的内容结构混乱时，应先按上述顺序重组，再输出或写入文档。

## 9. Git 工作流

- `main` 是稳定分支，功能和修复从 `main` 创建独立分支并通过 PR 合并。
- 禁止直接向 `main` 提交功能改动，禁止向 `main` 强制推送。
- Commit 遵循 Conventional Commits，例如 `feat:`、`fix:`、`refactor:`、`docs:`、`test:`、`chore:`。
- 保持原子提交，一个 Commit 只解决一个关注点。
- 提交前检查完整暂存差异、文件清单和敏感信息，并运行对应验证。

若当前项目已有更具体的 Git 决策或用户给出明确提交安排，以对应决策和当前指令为准。

## 10. 交付前检查

交付前确认：

- [ ] 实现满足当前任务的验收标准。
- [ ] 没有引入需求之外的功能。
- [ ] 没有重复已有能力或引入不必要依赖。
- [ ] 函数、类和文件职责清楚，规模合理。
- [ ] 错误和外部输入得到显式处理。
- [ ] 没有硬编码或输出敏感信息。
- [ ] 差异只包含必要修改。
- [ ] 测试、编译和静态检查已通过。
- [ ] 汇报先结论后依据，并说明验证结果和剩余风险。
