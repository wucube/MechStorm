# 语言

审查中使用的精确术语。使用这些术语，不要用近义词。

## 名词

| 术语         | 含义                                                                 | 不使用               |
| ------------ | -------------------------------------------------------------------- | -------------------- |
| module       | 一个有界代码单元，具有接口和实现。不暗示特定的粒度或封装机制。       | component, service, unit |
| interface    | 模块暴露的表面——类型、函数签名、配置形状。                           | API, signature       |
| implementation | 接口背后的代码——模块的内部机制。                                  | —                    |
| depth        | 接口的简单性与实现的复杂性之比。深模块有简单接口和复杂实现。         | —                    |
| seam         | 模块之间的边界，实现可以在那里被替换。由至少两个适配器证明合理。     | boundary             |
| adapter      | 跨接缝的特定实现——例如 HTTP 适配器、内存适配器、mock 适配器。        | —                    |
| leverage     | 通过接口获得的杠杆——少量接口表面积控制大量实现。                     | —                    |
| locality     | 变更集中的程度——高局部性意味着变更集中在少数模块。                   | —                    |

## 形容词

| 术语     | 含义                                 |
| -------- | ------------------------------------ |
| deep     | 简单接口，复杂实现。杠杆高。       |
| shallow  | 接口几乎与实现一样复杂。杠杆低。   |

## 动词

| 术语       | 含义                                         |
| ---------- | -------------------------------------------- |
| deepen     | 通过简化接口和/或丰富实现来增加模块深度。   |
| leak       | 内部知识通过接口逸出。                       |

## 用法规则

1. **使用精确术语。** "模块是浅层的"比"组件做得太少"更精确。
2. **不要用近义词替代。** 术语表中的每个术语都有特定含义。用"组件"代替"模块"会丢失精度。
3. **术语是可组合的。** "定价逻辑通过接缝泄漏"组合了三个术语来表达一个精确的架构诊断。
4. **不使用时不要发明术语。** 如果术语表中的术语适用，就用它。如果都不适用，先考虑是否可以用现有术语的组合来表达，再发明新术语。
